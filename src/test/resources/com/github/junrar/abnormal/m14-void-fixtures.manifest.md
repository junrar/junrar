# M1.4 (issue #18) FirstWinDone distance-into-void fixtures — provenance & UnRAR oracle

Generated by `generate_m14_void_fixtures.py` (same directory). Each fixture drives a
*first-window void* match: a copy whose distance exceeds the current output position
`UnpPtr` while the first window pass has not completed (`FirstWinDone == false`). unrar
7.0.3+ (commit `5faaa45`) zero-fills such a run deterministically:

- `unpackinline.cpp` `CopyString`: `if (Distance>UnpPtr) { SrcPtr+=MaxWinSize; if (Distance>MaxWinSize || !FirstWinDone) { fill Length bytes with 0; return; } }`
- `unpack15.cpp` `CopyString15`: `if (!FirstWinDone && Distance>UnpPtr || Distance>MaxWinSize) { fill Length bytes with 0; }`

junrar's three legacy copyString else-branches instead wrap the source pointer with
`& MAXWINMASK` (`window[unpPtr] = window[destPtr++ & MAXWINMASK]`). The wrapped source
starts in the never-written high window region (zero) but, once `destPtr` increments past
window index 0, reads the member's OWN earlier literals — non-zero bytes unrar never emits.

**Void construction (all three engines).** The member writes N non-zero literals
(`UnpPtr = N`), then a match with `distance = N + 1` (`Distance > UnpPtr`, one past the
written region) and `length L > 1`. The wrap-copy emits `window[MAXWINSIZE-1] = 0` for the
first byte, then `window[0], window[1], …` for the rest; unrar zero-fills all L bytes.
`Distance > MaxWinSize` (0x400000) is unreachable in v15/v20 (max encodable distance
~64 KB / ~1 MB) and in this small v29 member, so every engine exercises the
`!FirstWinDone && Distance>UnpPtr` arm of the guard, not the `Distance>MaxWinSize` arm.

Corrupt-variant class per PARITY_PLAN.md §4.3(1) ("a corrupt archive is corrupt however it
was made"). No compressor or rar/unrar binary is committed. Oracle = UnRAR 7.23
(`unrar x -kb`, keep-broken: each member fails the data-CRC check — expected for a corrupt
archive — and the broken member is kept for byte inspection).

The v15 patch touches only raw distance bits (no Huffman symbol), so the rest of the
bitstream stays byte-aligned. Coordinates were located by temporary instrumentation
(a `DBG_absShift` counter in `BitInput` incremented at each `unpReadBuf` memmove, plus a
print in `Unpack15.shortLZ`'s `Length==14` branch); the instrumentation was reverted — no
`src/main` file changed in the committed tree.

## UnRAR 7.23 observed behaviour

`unrar x -kb -y` exits `3`; the byte-patched/void member fails with `- checksum error`
(`Total errors: 1`). With `-kb` the broken member is kept:

- `void-dist-v20.rar` / `void-dist-v29.rar`: member (`void20` / `void29`, 6 bytes) is
  `41 41 41 00 00 00` — `AAA` then the length-3 void zero-filled.
- `void-dist-v15.rar`: `BOATMO~1.WAV` is 56464 bytes; the length-5 void at output offset
  34759 is `00 00 00 00 00`; sibling `LICENSE.TXT` extracts `OK`.

## junrar behaviour (RED on current code)

`UnpackDistanceIntoVoidTest` extracts each corrupt member through the public `Archive`
API (bytes are produced before the expected data-CRC `RarException`) and asserts byte
parity with the oracle. On current junrar the produced output leaks the wrapped window
bytes instead of zeros, so every assertion fails:

- v20 / v29: junrar emits `41 41 41 00 41 41` (leaks the member's own `A A`) vs oracle
  `41 41 41 00 00 00`.
- v15: junrar emits `00 52 49 46 46` (NUL + `RIFF`, the WAV's own header from
  `window[0..3]`) at offset 34759 vs oracle `00 00 00 00 00`.

Once the FirstWinDone guard lands in `Unpack15`/`Unpack20`/`Unpack` copyString, junrar
zero-fills identically and all three assertions pass (GREEN).
